1. PITCH EN UNE PHRASE
-----------------------
CTF de post-exploitation Linux : a partir d'un acces SSH initial obtenu
par phishing, le joueur remonte toute une chaine de privilege escalation
(10 flags) jusqu'a l'exfiltration finale de donnees chiffrees.

2. LE SCENARIO (histoire racontee au joueur)
---------------------------------------------
- Client fictif : Meridian Capital (societe de gestion d'actifs).
- Une autre "equipe" a deja fait le phishing/l'ingenierie sociale et
  recupere les identifiants SSH d'un stagiaire IT : j.martin.
- Le joueur prend le relais a partir de cet acces initial (c'est donc
  un exercice de POST-exploitation, pas d'acces initial).
- Objectif : escalader de compte en compte jusqu'a root, puis exfiltrer
  et dechiffrer les donnees sensibles.
- Connexion : ssh j.martin@<host> -p <port> / mot de passe Welcome2024!

3. POURQUOI CES CHOIX TECHNIQUES (questions pieges probables)
----------------------------------------------------------------
Q: Pourquoi un seul conteneur Docker et pas plusieurs machines
   (poste / serveur / DC) ?
R: Le plugin CTFd Docker gere "une image = un conteneur". Simuler
   plusieurs hotes aurait demande du Docker-in-Docker privilegie
   (risque de securite) ou un support compose incertain. A la place,
   la segmentation reseau est remplacee par une SEGMENTATION PAR
   PRIVILEGE (comptes Unix distincts, sudo cible, capabilities) :
   le mouvement lateral "inter-comptes" sur un meme host est une
   competence aussi realiste que le pivot reseau, et bien plus fiable
   a faire tourner en live pendant une soutenance.

Q: Pourquoi ne pas avoir monte le vrai socket Docker de l'hote pour
   le flag 8 (au lieu d'un faux service maison) ?
R: Exposer /var/run/docker.sock dans le conteneur donnerait un accès
   root sur la machine HOTE a quiconque compromet le conteneur -
   inacceptable en environnement mutualise. A la place : un daemon
   "orchestrator.py" invente en interne, avec une vraie vulnerabilite
   (deserialisation Python non securisee) qui donne la meme sensation
   pedagogique sans aucun risque d'evasion vers l'hote reel.

Q: Pourquoi pas de vulnerabilite web ?
R: Contrainte projet : ne pas chevaucher le challenge web d'un
   collegue. Ce CTF est volontairement 100% systeme : permissions
   Unix, cron, SUID, sudo, capabilities, un service interne sur
   socket UNIX (pas HTTP), + crypto appliquee a la fin.

Q: Les flags sont-ils les memes a chaque partie (statiques) ?
R: Oui, acceptable car chaque equipe recoit sa propre instance Docker
   isolee (reseau non partage, detruite apres usage) : pas de risque
   qu'une equipe vole le flag d'une autre. Une evolution possible
   (non implementee) serait de regenerer les flags a partir d'une
   variable d'environnement (TEAM_ID) injectee au demarrage.

4. LA CHAINE COMPLETE (a réciter dans l'ordre)
------------------------------------------------
j.martin (SSH, mdp fourni)
  -> F1 lecture fichier dans le home
  -> F2 enumeration de /var/backups (fichiers .bak, leurres inclus)
  -> F3 mot de passe de svc_backup trouve dans .bash_history
su svc_backup
  -> F4 cron root inscriptible par le groupe svc_backup -> mdp r.dubois
su r.dubois (groupe "analysts")
  -> F5 binaire SUID logviewer vulnerable a l'injection de commande
  -> F6 sudo NOPASSWD sur "find" (GTFOBins) -> shell app_agent
sudo -u app_agent find . -exec /bin/sh \;
  -> F7 binaire avec capability cap_dac_read_search -> lecture fichiers
     arbitraires (recupere aussi un token d'orchestrateur)
  -> F8 socket UNIX orchestrator.sock + token -> deserialisation
     Python (pickle) -> RCE en tant que ROOT -> exfiltration de
     vault.zip, final.gpg, pin.hash
(root atteint via l'orchestrateur, en local, pas de shell root direct)
  -> F9 cassage du mot de passe de vault.zip (zip2john + john,
     wordlist rockyou.txt) -> "iloveyou"
  -> F10 cassage du PIN 6 chiffres (hashcat, attaque par masque) ->
     dechiffrement GPG de final.gpg -> flag final

Regle importante : chaque flag est valide independamment dans CTFd,
pas besoin d'avoir fini toute la chaine pour scorer les premiers.

5. DETAIL FLAG PAR FLAG (technique + commande cle + pourquoi ca marche)
--------------------------------------------------------------------------
F1 (50 pts) - Reconnaissance
  cat ~/welcome_note.txt
  -> Un fichier laisse dans le home du nouvel arrivant.

F2 (75 pts) - Enumeration filesystem
  grep -r "flag" /var/backups/
  -> Sauvegardes mal nettoyees, plusieurs .bak dont des leurres.
     Le fichier legitime mentionne aussi les comptes internes
     (svc_backup, r.dubois, app_agent) -> indice pour la suite.

F3 (100 pts) - Recolte d'identifiants
  cat ~/.bash_history
  -> Le mot de passe de svc_backup a ete tape par erreur dans le
     shell de j.martin (frappe au mauvais prompt) et reste en clair
     dans l'historique. Classique : ne JAMAIS taper un mot de passe
     en argument de commande.

F4 (125 pts) - Abus de cron
  cat /etc/cron.d/meridian  puis  ls -la /opt/scripts/cleanup.sh
  -> Script execute par root toutes les minutes, mais modifiable par
     le groupe svc_backup (rwxrwxr-x). On y ajoute une commande qui
     copie le mot de passe de r.dubois vers /tmp, on attend le cron.
  -> Principe : si un script lance par un compte plus privilegie est
     ecrit par un compte moins privilegie, c'est une escalade directe.

F5 (150 pts) - SUID vulnerable
  find / -perm -4000 -type f 2>/dev/null
  -> logviewer, SUID root, reserve au groupe "analysts". Il construit
     une commande shell "cat /var/log/meridian/<arg>.log" sans nettoyer
     l'argument -> injection de commande :
     logviewer "app; cat /root/flag5.txt #"
  -> Principe : un binaire SUID qui appelle system()/popen() avec une
     entree utilisateur non filtree = injection + privilege root.

F6 (175 pts) - Sudo mal configure / GTFOBins
  sudo -l
  -> r.dubois peut lancer /usr/bin/find en tant qu'app_agent sans mdp.
     find a une fonction -exec qui lance un shell :
     sudo -u app_agent find . -exec /bin/sh \;
  -> Principe : GTFOBins recense les binaires "detournables" via sudo
     (find, vim, less, awk, etc.) pour obtenir un shell.

F7 (200 pts) - Linux capabilities
  getcap -r / 2>/dev/null
  -> py-agent possede cap_dac_read_search=ep : capacite qui contourne
     TOUTES les verifications de lecture/traversee de repertoire,
     meme sans etre root. Permet de lire /root/flag7.txt et un token
     d'orchestrateur, sans avoir besoin de SUID ni de sudo.
  -> Principe : les "capabilities" Linux decoupent les pouvoirs de
     root en unites fines ; en donner une seule (par erreur) peut
     suffire a tout compromettre.

F8 (225 pts) - Deserialisation non securisee (RCE root)
  -> Un service "orchestrator.py" ecoute sur un socket UNIX
     (/run/meridian/orchestrator.sock), protocole JSON minimal. La
     commande restore_config prend un payload base64 passe TEL QUEL
     a pickle.loads() -> execution de code arbitraire cote serveur
     (qui tourne en root). On fabrique un objet Python dont la methode
     __reduce__ renvoie os.system(<commande>) : sa desererialisation
     execute la commande. On exfiltre vault.zip, final.gpg, pin.hash.
  -> Principe : ne JAMAIS deserialiser du pickle (ou equivalent) venant
     d'une source non fiable : la deserialisation peut executer du code.

F9 (250 pts) - Cassage offline (zip)
  zip2john vault.zip > vault.hash
  john --wordlist=rockyou.txt vault.hash
  -> Mot de passe faible ("iloveyou") trouve par dictionnaire.
  -> Principe : un chiffrement fort avec un mot de passe faible
     reste casse trivialement hors-ligne.

F10 (300 pts, final) - Attaque par masque + GPG
  hashcat -m 1400 -a 3 pin.hash ?d?d?d?d?d?d
  -> pin.hash = SHA-256 d'un code PIN a 6 chiffres : espace de
     recherche minuscule (1 million de combinaisons), casse quasi
     instantanement par une attaque par masque (uniquement des chiffres,
     pas un dictionnaire generique).
  -> Le PIN trouve sert de passphrase pour dechiffrer final.gpg
     (chiffrement symetrique GPG) -> flag final.
  -> Principe : un "PIN numerique" est un espace de mots de passe
     ridiculement petit face a hashcat, contrairement a un mot de
     passe alphanumerique complexe.

7. CHIFFRES A RETENIR
-------------------------
- 10 flags, 1650 points au total (50 a 300 points, croissant).
- Format des flags : MERIDIAN{...}
- Une seule instance Docker (bouton Start dans CTFd), tout se joue
  dessus du flag 1 au flag 10.
- Identifiant de depart : j.martin / Welcome2024!
- Outils de cassage utilises : zip2john, john the ripper (rockyou.txt),
  hashcat (mode 1400 = SHA-256, attaque par masque -a 3).

9. PHRASE DE CONCLUSION POUR LA SOUTENANCE
------------------------------------------------
"Ce challenge illustre qu'une compromission complete ne demande pas
forcement de faille 0-day : une succession de petites erreurs de
configuration et d'hygiene (mot de passe dans un historique, cron
inscriptible, SUID mal ecrit, sudo trop permissif, capability oubliee,
deserialisation non securisee, mots de passe faibles) suffit a mener
d'un simple acces stagiaire jusqu'a la compromission totale du systeme."
